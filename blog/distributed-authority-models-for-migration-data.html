Distributed authority models for migration data governance

The UN Migration Network Hub is not a multi tenant SaaS with configuration tables for each agency. It is 64 distinct organizations — the IOM itself, United Nations entities like OCHA and UNEP, intergovernmental bodies such as OECD and World Bank, plus country level chapters from Argentina to Zambia all contributing different types of content under mutually agreed editorial contracts.

This coordination happens without a central database. Each agency owns its own instances while referencing shared concepts — refugees, forced displacement scenarios, climate migration events — through entity relationships that traverse institutional boundaries. The question is not how you replicate data across replicas; it is where authority lives for each piece of information and what rules enforce the boundary between who can author content versus who can see it.

The architecture pattern

We built this on Drupal 10, which gives us three primitives: sites are logical units with their own user repositories and role definitions; users belong to multiple of these administrative contexts without losing membership elsewhere; content types define what information exists while entity reference fields establish cross site connections. When an OCHA field officer uploads a displacement report that UNEP will later annotate with climate vulnerability metadata, the system enforces this workflow through permissions defined at each organizational boundary.

The distinction between authority (who can change data) and visibility (who can see it) is why access control cannot be an afterthought. It must exist in every design decision from content type definition onward.

Role based editorial contracts

I have worked across several different deployments where agencies requested a central repository to host all migration data for easier collaboration and better search results. The problem was that when you give one agency write access, they inherit authority over what others built there, which means the entire system's content model breaks whenever someone with elevated privileges makes changes without consulting affected teams.

The three layer permission pattern

Datum level (entity fields)

This is where a country chapter author can edit only their own displacement reports while leaving shared vocabulary tables untouched. Each entity type defines its own field permissions — the metadata schema for refugee flows differs from editorial notes on humanitarian corridors.

Type level (content types and bundles)

An intergovernmental body might have write access to all displacement scenario entities but not migration statistics, since national statistical offices guard their methodologies. Within content types themselves you can define subtypes that further partition authority — an OCHA user could manage situation reports within the broader humanitarian intelligence model without affecting policy documents maintained by IOM headquarters.

Suite level (sites and administrative domains)

This is where each member state defines their own organizational boundaries, whether as separate Drupal instances federated through entity API calls or modules installed across multiple systems. A World Bank economist reviewing global migration statistics might only have visibility into datasets marked for public consumption while the internal analytical workspaces remain isolated to agencies that classified them there.

Educating stakeholders on permission semantics

The hardest conversations involve convincing technical directors from partner organizations why we cannot just give everyone admin access so they can fix whatever is broken. When an organization understands the distinction between read write and full control, their requests become more realistic about what permissions are necessary for specific tasks.

Cross site entity relationships

The UN Migration Network Hub uses custom modules to extend Drupal's built in cross instance linking without duplicating data. An OCHA displacement report references a migration corridor defined by IOM while UNEP adds climate context from their own content type, all resolved through API endpoints that respect permission boundaries at each hop.

Maintaining coherence across 64 instances

This coordination happens without imposing uniformity. Some agencies operate on Pantheon cloud environments; others prefer bare metal infrastructure in their sovereign data centers with custom caching layers for performance optimization during peak reporting periods. When a new content type schema lands at IOM headquarters, we provide migration paths that allow each partner to adopt it according to their own development velocity while contributing enhancements back through pull requests organized within the shared codebase.

Uniformity becomes coercion if you demand identical versions across all deployments without understanding why particular agencies need different capabilities. Flexibility means providing multiple implementation patterns for common requirements and letting organizations choose what fits their operational model.

The editorial contract mechanism

We document these relationships through configuration files that encode who maintains each content type, how cross references get resolved during publishing, and which agency signs off on changes to shared schemas. An OCHA analyst submitting a displacement report for Syria must reference an existing migration corridor entity; the system will reject submissions attempting to create corridors without approval from IOM's editorial board.

Caching across distributed systems

We use Redis clusters configured per institutional boundary with shared read caches that respect permission sets. When OCHA queries displacement statistics for a particular corridor, their session reads cached data while write operations require fresh API calls routed to the owning agency's instance. This pattern scales without compromising authority boundaries.

The next decade

As we migrate from Drupal 10 toward AI integrated content workflows, these permission patterns remain foundational for multi organizational systems where autonomous teams need different capabilities over shared information resources. The question is no longer whether institutions can coordinate without a central authority; it is how you design systems that survive contact with multiple independent operators while maintaining coherent global views of migration data.
