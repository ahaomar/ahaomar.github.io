Why governance is the first thing you design, not the last

Every multi agency platform I have built starts with the same wrong question from stakeholders: which CMS should we use. It is the wrong first question because by the time you are choosing software, you have already skipped the decision that determines whether the platform survives contact with real institutions.

The decision that matters first is governance. Who owns which piece of content. Which agency can publish without review, and which cannot. What happens when two UN entities disagree about how a dataset should be labeled. Who has the authority to take a page down. None of that is a technical question, and none of it can be retrofitted cheaply once forty organizations are already using the system.

What this looked like on the UN Migration Network Hub

When I architected the UN Migration Network Hub, the platform needed to serve 195 member states and dozens of UN agencies inside a single Drupal 10 ecosystem, without any single agency feeling like a guest in someone else's system. That is a governance problem before it is an infrastructure problem.

The technical answer was a federated content and permission model: every agency operates inside clearly scoped boundaries, with its own editorial workflow, but all of it sits on shared infrastructure that stays secure and consistent for 195 member states at once. Building that model required sitting with the actual governance structure of the Global Compact for Migration before a single content type was defined, and then encoding that structure directly into the permission system rather than bolting it on afterward.

A platform that gets the technology right and the governance wrong does not fail slowly. It fails the first time two stakeholders disagree about who is allowed to do what.

The pattern repeats everywhere I have seen it

The same pattern showed up when I built the CLIMB platform for IOM and the Platform on Disaster Displacement. The hard part was never rendering GIS layers or building the data pipeline, real as those problems were. The hard part was designing a data model that let policy experts trust the provenance of every figure a G7 delegation might cite, which is a governance requirement disguised as a data engineering one.

If you are architecting a platform for an institution rather than a company, treat the org chart, the approval chain, and the accountability structure as your first spec document. The CMS choice, the hosting decision, the caching layer: all of that follows from getting the governance model right, and none of it can compensate for getting it wrong.
